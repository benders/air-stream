API - PurpleAir

A RESTful API for PurpleAir sensors

Getting Started

Welcome to the PurpleAir API

PurpleAir was founded on principles of openness, sharing, and community. It is to strengthen these principles and ensure the success of our entire community that we put together updated terms of service, data license, and data attribution requirements when using PurpleAir data and this API.

Please take a moment to review them and note the attribution guide and data license agreement. If you have any questions or need more information, we have an excellent resource at https://community.purpleair.com/c/data/api/.

Basic Principles

Providing a way to programmatically interact with PurpleAir data, the PurpleAir Sensor API is based on REST principles employing the following HTTP methods:
GET: Returns information.
POST: Creates entries.
PUT: Updates entries.
DELETE: Deletes entries.

SSL is required to access this API. If the URL you use in requests to the server does not start with https:// you will get an error.

Getting Started - Using API Keys

API keys are required to interact with the PurpleAir API, and must be included in every API request.


Creating API Keys

The first step to obtaining API keys is creating an account on the Developer Dashboard by logging in with a Google or Google-Associated email.

If using a Google or Google-Associated email is not an option, send an email to contact@purpleair.com for assistance.

Among other things, the dashboard allows you to:

  • Create and manage your API keys.

  • Monitor API usage by points consumed, data used, or number of calls made.

  • Examine logs for your calls using the Logs Explorer.

  • View your current points balance and purchase additional points.


When creating an API key, you must select its type. There are two types of keys:

  • READ keys are used when reading entries with a GET request.
  • WRITE keys are used when creating and deleting groups or group members with POST or DELETE request.

Supplying the wrong type of API key in your requests will result in an ApiKeyTypeMismatchError.


Using API Keys

Every API request must include your API key. You can do so using one of three ways:

  • As an HTTP-Header * Preferred.
  • As a QueryString parameter in the URL.
  • Included in the JSON body of a POST request.

Each endpoint has a section labeled "Send a Sample Request," which allows you to quickly test out an endpoint.
Try using the documentation below to send a request to "Check an API Key" and "Get Organization Data" with your newly created API read key.


Making Successful Requests

Rate Limiting

To protect the API, keys are rate limited. This means that there is a maximum rate at which the server will accept your requests. Exceeding the rate limit by sending requests too quickly will result in a RateLimitExceededError.

The maximum rate you can send requests depends on which endpoint you are querying. This is listed in each endpoint's description as the 'default rate limit.' As performance improvements are made to the API, default rate limits may be adjusted to enable more frequent requests.

PurpleAir can adjust rate limiting per API key. If default rate limits impact your use case, please send an email to contact@purpleair.com for assistance.


Frequent Problems and Troubleshooting

  • Ensure your API request is properly formatted. Failing to do so may result in an error being returned or unexpected behavior.
  • Your organization account must have the neccessary amount of API points for each request you send.
    • Your organization starts with 1 million free points to help you get started.
    • Free points are available for sensor owners as described in our API Points for Sensor Owners community article.
    • Additional points are available for purchase with more details found in our API Pricing community article.
  • If you send requests too quickly, you will receive a RateLimitExceededError.

If you have any problems, the Developer Dashboard's Logs Explorer can be used to examine requests you've sent and errors encountered.

You can also create a community post at https://community.purpleair.com/c/data/api/ for assistance from PurpleAir and other community members.

Configurable Decimal Places
You can control how many decimal places are returned for each numeric field by using the syntax: fieldname|dN, where N is the maximum number of decimal places to include.

For example, if pm2.5_atm_a is recorded as 3.23:

Requested Field Returned Value
pm2.5_atm_a|d0 3
pm2.5_atm_a|d1 3.2
pm2.5_atm_a|d2 3.23
pm2.5_atm_a|d3 3.23

Note: No trailing zeros are added if the requested number of decimal places exceeds those present in the stored value. Requesting pm2.5_atm_a|d3 in the table above returns 3.23 rather than 3.230, because 3.23 is the actual value stored in the database.

As decimal places are configurable per field, varying precision levels can be applied to different fields in the same request:
fields=temperature|d1,pm2.5_atm_a|d2,pm2.5_atm_b|d0

The ALT Variant estimated mass concentration PM2.5 (g/m) is derived from the particle counts. PM2.5 are fine particulates with a diameter of fewer than 2.5 microns.

pm2.5_alt returns average ALT variant for channel A and B but excluding downgraded channels.
pm2.5_alt_a returns channel A ALT variant.
pm2.5_alt_b returns channel B ALT variant.

The following method is used to calculate the ALT variant:

Step 1: Calculate the number N of particles (per deciliter) in each of the size categories 0.3-0.5 um, 0.5-1 um, and 1-2.5 um. Note: since the Plantower output provides the total number of particles >0.3 um, the total number >0.5 um, etc., one must subtract the total number >0.5 um from the total number > 0.3 um to arrive at the number of particles in the 0.3-0.5 um size category. Similar subtractions will result in three size categories under 10 um: 0.3-0.5 um, 0.5-1 um, 1-2.5 um.

Step 2: Select an average diameter D for each of the three size categories up to 2.5 um. For example, the average particle diameter D in the 0.3-0.5 um size category category must be between 0.3 and 0.5 um, so we can approximate it by the midpoint (0.4 um) or by the geometric mean of the size boundaries (0.387 um). Several manufacturers use the geometric mean so we use it throughout this work.

Step 3: Calculate the total particle volume V in each size category. This is given by multiplying the number of particles N times the volume per particle: V = N D 3 / 6

Step 4: Multiply by a density to arrive at an estimate of the mass concentration M in each size category. We choose to be the density of water, a choice that is also made by the Model 3330 manufacturers: M = V

Step 5: Add the three mass concentrations to estimate PM2.5.

Step 6: Multiply by a calibration factor (CF) based on comparisons with nearby or collocated reference monitors. Based on comparisons of two PurpleAir monitors with collocated reference monitors (e.g., the SidePak and Piezobalance) in 124 experiments since January 2019, a CF of 3 is recommended. 

const cf_3 = 3.0; 
function pm2_5_alt(p0_3, p0_5, p1_0, p2_5, cf) { 
    var _cf = parseFloat(cf); 
    if (isNaN(_cf)) _cf = cf_3; 
    var m1 = getMass(p0_3 - p0_5, 0.3,  0.5); 
    var m2 = getMass(p0_5 - p1_0, 0.5,  1.0); 
    var m3 = getMass(p1_0 - p2_5, 1.0,  2.5); 
    var pm2_5 = m1 + m2 + m3; 
    return pm2_5 * _cf; 
} 
function getMass(countDl, sizeLower, sizeUpper) { 
    var densityGCm3 = 1.0; 
    var geometricMeanUm = Math.sqrt(sizeLower * sizeUpper); 
    var volume = countDl * Math.PI * Math.pow(geometricMeanUm, 3) / 6.0; 
    return volume * densityGCm3 / 100.0; 
}

Configurable Decimal Places
You can control how many decimal places are returned for each numeric field by using the syntax: fieldname|dN, where N is the maximum number of decimal places to include.

For example, if pm2.5_atm_a is recorded as 3.23:

Requested Field Returned Value
pm2.5_atm_a|d0 3
pm2.5_atm_a|d1 3.2
pm2.5_atm_a|d2 3.23
pm2.5_atm_a|d3 3.23

Note: No trailing zeros are added if the requested number of decimal places exceeds those present in the stored value. Requesting pm2.5_atm_a|d3 in the table above returns 3.23 rather than 3.230, because 3.23 is the actual value stored in the database.

As decimal places are configurable per field, varying precision levels can be applied to different fields in the same request:
fields=temperature|d1,pm2.5_atm_a|d2,pm2.5_atm_b|d0

pm2.5_alt Number

The ALT Variant estimated mass concentration PM2.5 (g/m) is derived from the particle counts. PM2.5 are fine particulates with a diameter of fewer than 2.5 microns.

pm2.5_alt returns average ALT variant for channel A and B but excluding downgraded channels.
pm2.5_alt_a returns channel A ALT variant.
pm2.5_alt_b returns channel B ALT variant.

The following method is used to calculate the ALT variant:

Step 1: Calculate the number N of particles (per deciliter) in each of the size categories 0.3-0.5 um, 0.5-1 um, and 1-2.5 um. Note: since the Plantower output provides the total number of particles >0.3 um, the total number >0.5 um, etc., one must subtract the total number >0.5 um from the total number > 0.3 um to arrive at the number of particles in the 0.3-0.5 um size category. Similar subtractions will result in three size categories under 10 um: 0.3-0.5 um, 0.5-1 um, 1-2.5 um.

Step 2: Select an average diameter D for each of the three size categories up to 2.5 um. For example, the average particle diameter D in the 0.3-0.5 um size category category must be between 0.3 and 0.5 um, so we can approximate it by the midpoint (0.4 um) or by the geometric mean of the size boundaries (0.387 um). Several manufacturers use the geometric mean so we use it throughout this work.

Step 3: Calculate the total particle volume V in each size category. This is given by multiplying the number of particles N times the volume per particle: V = N D 3 / 6

Step 4: Multiply by a density to arrive at an estimate of the mass concentration M in each size category. We choose to be the density of water, a choice that is also made by the Model 3330 manufacturers: M = V

Step 5: Add the three mass concentrations to estimate PM2.5.

Step 6: Multiply by a calibration factor (CF) based on comparisons with nearby or collocated reference monitors. Based on comparisons of two PurpleAir monitors with collocated reference monitors (e.g., the SidePak and Piezobalance) in 124 experiments since January 2019, a CF of 3 is recommended. 

const cf_3 = 3.0; 
function pm2_5_alt(p0_3, p0_5, p1_0, p2_5, cf) { 
    var _cf = parseFloat(cf); 
    if (isNaN(_cf)) _cf = cf_3; 
    var m1 = getMass(p0_3 - p0_5, 0.3,  0.5); 
    var m2 = getMass(p0_5 - p1_0, 0.5,  1.0); 
    var m3 = getMass(p1_0 - p2_5, 1.0,  2.5); 
    var pm2_5 = m1 + m2 + m3; 
    return pm2_5 * _cf; 
} 
function getMass(countDl, sizeLower, sizeUpper) { 
    var densityGCm3 = 1.0; 
    var geometricMeanUm = Math.sqrt(sizeLower * sizeUpper); 
    var volume = countDl * Math.PI * Math.pow(geometricMeanUm, 3) / 6.0; 
    return volume * densityGCm3 / 100.0; 
}
